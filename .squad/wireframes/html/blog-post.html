Building Accessible React Components Without Slowing Down Shipping

Every team says they care about accessibility until a release gets tight. Then semantics, focus order, and states become “follow-up work.” The pattern that finally worked for us was simple: move the decisions that matter into the component contract, then make the happy path the accessible path.

Accessible components should feel faster to use than inaccessible ones — for the implementer, reviewer, and the person navigating with a keyboard.

Designing the contract

We started by listing the decisions that should never be deferred to product teams: button semantics, icon labeling, error messaging slots, and keyboard behavior for composite widgets. Once those decisions lived inside primitives, feature teams spent less time remembering rules and more time composing real interfaces.

The payoff was strongest in states-heavy components such as menus, dialogs, and comboboxes. Instead of shipping loose wrappers, we defined opinionated primitives with semantic defaults, and then exposed only the extension points that design actually needed.

CalloutBox Callout · practical heuristic

If a reviewer needs a checklist to verify the default state, the component API is probably too loose. Good primitives make misuse slightly inconvenient.

Guardrails first

We built guardrails around the moments where teams usually regress:

  • automatic association between labels, help text, and error text;
  • focus-visible styling that matches the design system in both light and dark themes;
  • sensible fallbacks for icon-only actions and loading states;
  • slot structures that encourage headings and landmarks in the right order.
CodeBlock
Button.tsx tsx 
export function Button({
  intent = "primary",
  icon,
  children,
  ...props
}) {
  const needsAriaLabel = icon && !children;

  return (
    <button
      data-intent={intent}
      aria-label={needsAriaLabel ? "Open menu" : props["aria-label"]}
      {...props}
    >
      {icon}
      {children}
    </button>
  );
}
// Encodes an accessible default instead of relying on downstream memory.

Composable APIs

We resisted “do everything” components and focused on composition. A good primitive should be easy to combine with layout and content systems, but still strict about semantics. That usually meant separating structure from polish: primitives handled role, state, and keyboard logic; wrappers layered on spacing, art direction, and editorial rhythm.

That division also helped us explain where client boundaries actually belong. Some interactions genuinely need JavaScript, like menus or reading progress. Others only look dynamic but can stay server-rendered. Keeping that line sharp protected performance and reduced hydration cost.

CalloutBox Warning · common trap

Adding client logic “just in case” is how content-heavy pages become sluggish. If an interaction is decorative, prefer CSS. If it changes focus, state, or layout semantics, earn the boundary with intent.

Testing rhythm

Our testing rhythm blended accessibility checks into ordinary UI review instead of making them a separate gate. Designers validated spacing and contrast, engineers verified keyboard flows, and QA spot-checked screen-reader labels on the highest-risk paths.

Surface Guardrail Review cue
Buttons Accessible name required for icon-only use Keyboard focus visible in dark mode
Inputs Descriptions and errors linked automatically Tab order matches visual flow
Dialogs Focus trapped and returned on close Escape closes without losing context

What shipped

In the end, the biggest win wasn’t the component library itself. It was the calmer review process. With semantics and focus handled centrally, teams could talk about clarity, hierarchy, and product tradeoffs instead of relitigating baseline accessibility each sprint.

That’s the kind of system work I want from a blog: fewer declarations, more honest implementation details, and enough specificity that another developer could borrow the pattern tomorrow.

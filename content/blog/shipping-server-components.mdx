## Start with the quiet version

Server Components work best when the initial version of a page is already useful before JavaScript takes over. That framing changes how you design the experience.

### What stays on the server

Anything that can be rendered from content, route params, or static configuration belongs on the server first. It keeps the page smaller and the mental model cleaner.

## Add islands only for interaction

Once the server-rendered version is solid, I add client islands only where interaction genuinely improves the experience.

### Good client candidates

- URL-driven filters
- Reading progress indicators
- Table of contents scroll-spy behavior

```tsx
export async function generateMetadata() {
  return {
    title: "Server-first page",
    description: "A page that earns its interactivity",
  };
}
```

<Callout tone="success" title="A healthy bias">
I try to make the static version pleasant first. Interactivity should add precision, not rescue a weak layout.
</Callout>

## Performance is part of the aesthetic

Fast pages feel more composed. When a page renders quickly and moves with intention, the interface reads as more trustworthy.

### The trade-offs I watch

| Decision | Risk | Guardrail |
| --- | --- | --- |
| Client search params | Static bailout | Keep islands wrapped and intentional |
| Heavy animations | Jank | Prefer transform and opacity |
| Overfetching data | Slow TTFB | Fetch once and reuse in metadata + page |

## The frontend payoff

A server-first setup leaves more room for the details that actually matter: stronger typography, cleaner information hierarchy, and micro-interactions that feel earned.

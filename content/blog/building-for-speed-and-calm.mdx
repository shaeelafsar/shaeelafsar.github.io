## Start with the rendering model

When a project has a lot of content, page composition, and mostly-static data, Server Components are a strong default. They simplify data flow and reduce the amount of client JavaScript needed to get useful UI on screen.

### Keep client islands intentional

Interactivity still matters, but it should be scoped to the places that truly need it: menus, filters, timelines, forms, and reading tools.

### A good boundary lowers maintenance costs

Each client component introduces more hydration work, more testing paths, and more state management decisions. That cost is worth paying when the interaction improves the product.

## Performance is also a communication problem

Teams move faster when they can explain why a page feels slow and what tradeoffs they are making.

### Prefer understandable wins

- Reduce client bundles before micro-optimizing animation code.
- Avoid needless state synchronization across distant components.
- Make loading and error states first-class parts of the design.

### Ship feedback loops early

A stable build, clear linting, and realistic sample content make it easier to catch layout and content problems before launch.

## Calm products feel faster

Perceived performance is shaped by rhythm, continuity, and confidence. Good defaults, strong placeholders, and coherent state transitions make an interface feel more dependable.
